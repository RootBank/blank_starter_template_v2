# Collection Module Template — Overview

> A provider-agnostic template for integrating any payment provider with the Root Platform.

## Overview

This template provides the scaffolding to connect any payment provider to the Root Platform insurance system. There is no provider implementation pre-installed. Instead, you describe your provider via a spec file and the scaffold script generates the implementation. The architecture uses dependency injection, layered separation, and provider-agnostic interfaces so providers are interchangeable.

The module runs as a Root Platform collection module — deployed via the Root CLI. It handles payment method creation, payment lifecycle events, webhook processing, and dashboard rendering.

## Data Flow

### Lifecycle Hook Flow (Root → Module → Provider)

1. Root Platform calls an exported lifecycle hook (e.g. `renderCreatePaymentMethod`).
2. The hook resolves services from the DI container via `getContainer()`.
3. Services call clients (`{Provider}Client`, `RootClient`) to perform operations.
4. Adapters transform data between provider format and Root format.
5. The hook returns a response (HTML string, module data, or void).

### Webhook Flow (Provider → Module → Root)

1. Payment provider sends a webhook to `processWebhookRequest`.
2. `webhook-hooks.ts` verifies the signature using `providerWebhookSigningSecret`.
3. The event type is matched in a `switch` statement.
4. The matched case resolves a controller from the DI container.
5. The controller calls services and adapters to process the event.
6. `RootService` updates the Root Platform (payment status, policy data).

### DI Container Initialization

1. `code/main.ts` calls `getContainer()` on module load.
2. `container.setup.ts` registers all services with `ServiceToken` symbols.
3. All services default to `SINGLETON` lifetime (one instance per container).
4. Services resolve their dependencies through the container factory functions.

## Key Concepts

- **Provider-agnostic tokens**: `PROVIDER_CLIENT` and `PROVIDER_SERVICE` are generic DI tokens. Register your scaffolded provider's client and service under these tokens in `container.setup.ts` after running `scaffold:provider`.
- **Lifecycle hooks**: Exported functions that Root Platform calls at specific points (policy issued, payment created, payment method assigned, etc.). Stubs are in `lifecycle-hooks/`.
- **Webhook hooks**: A separate entry point for processing incoming webhooks from the payment provider. Implement signature verification and event routing in `webhook-hooks.ts`.
- **Adapters**: Pure transformation functions converting provider data to Root Platform format. They never make API calls.
- **Everything generated**: Provider-specific files (`{provider}-client.ts`, `{provider}.service.ts`, `{provider}-to-root-adapter.ts`, `{provider}-events.ts`) are generated by `npm run scaffold:provider`.
